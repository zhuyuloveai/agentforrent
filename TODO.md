## 评分机制分析与优化策略（2026-03-02 确认）

### 一、评分机制确认

#### 用例类型与分值
| 类型 | 常量 | 分值 | 评分逻辑 |
|------|------|------|---------|
| 聊天类 | Chat | 5分 | 返回结果即得分，不评判内容 |
| 单轮简单/复杂查询 | Single | 10-15分 | 按 houses 命中率给分 |
| 多轮复杂任务 | Multi | 20-30分 | 按 houses 命中率给分，写操作另行验证 |

#### 命中率评分原理
判题器在执行用例前调用 `/api/houses/init` 重置数据，然后用相同条件查询 API 得到"基准答案集"，再和 agent 返回的 `houses` 数组做对比：

```
得分 = 命中数 / 基准答案集大小 × 该用例满分
```

**核心结论：返回的每一套房必须满足用户所有条件，错返一套直接损失命中率。**

#### 写操作评分（双重验证）
1. **API 状态验证**：判题器查询 `/api/houses/{id}` 确认状态变更 → 不调工具直接0分
2. **houses 字段验证**：response 里 `houses` 数组必须包含被操作的房源 ID

#### 时间片消耗公式
```
t = 1 + max(0, (n/1000 - 1) × 0.3)  向上取整，n为单次调用token数
```
| token量 | 时间片 |
|---------|--------|
| ≤1000   | 1片    |
| 2000    | 2片    |
| 5000    | 3片    |
| 10000   | 4片    |

**总预算300片，得分相同时 token 消耗越少排名越高。**

**qwen3-32b 特别注意**：默认开启 thinking 模式，`<think>` 标签内容计入 token 消耗。一次含推理的调用可能产生 3000-8000 额外 thinking tokens，显著压缩可跑用例数量。

### 二、当前方案关键问题（按优先级）

#### P0：直接导致用例0分

**问题1：listing_platform 缺乏强制保障**
- rent/terminate/offline 接口 `listing_platform` 必填
- 当前仅在 tool description 里写"不确定时用安居客"，模型在多轮上下文下可能漏传
- 漏传 → API 400 → 写操作失败 → 用例0分
- **修复方案**：在 system prompt 里用独立规则强调，或在 `_execute_tool` 层补默认值兜底

**问题2：JSON 格式不合法**
- `temperature=1.0` 偏高，模型容易在 JSON 前后夹杂自然语言
- `_extract_json_from_text` 正则 `[^{}]*` 不支持嵌套，复杂输出会提取失败
- 解析失败 → response 无 houses 字段 → 用例0分
- **修复方案**：降低 temperature（建议0.3-0.5）+ 加强 prompt 格式约束

#### P1：直接影响命中率得分

**问题3：多轮对话筛选条件可能丢失**
- 第N轮只提取了新条件，之前的条件靠模型从 history 里"记忆"
- `Session.requirements` 字段设计了但未使用
- 7轮反复筛选场景（Multi 30分用例）风险极高
- **修复方案**：复活 requirements 字段，每轮从模型输出中提取/合并结构化需求

**问题4：MAX_TOOL_ROUNDS=3 超限后质量骤降**
- 超限后强制不带 tools 调用，模型只能"编"结果
- 复杂路径（查地标→查附近→查详情→租房）可能恰好需要4轮
- **修复方案**：根据场景调整上限（查询类3轮，含写操作类4-5轮）

#### P2：影响 token 效率与排名

**问题5：qwen3 thinking 模式未关闭**
- 当前 `chat_completion` 调用没有控制 thinking 参数
- 对于格式明确的查询任务，thinking 产生的 token 是纯浪费
- **修复方案**：在 system prompt 末尾添加 `/no_think` 或传 `enable_thinking=False` 参数（需确认 qwen3 API 支持方式）

**问题6：超出轮数后仍多调一次模型**
- 强制最终回复额外消耗 1 次模型调用
- **修复方案**：超限时直接从已收集的工具结果构建响应，跳过最后一次 LLM 调用

**实际输出示例**：
```
为您找到朝阳区24套房源，涵盖国贸、望京等热门商圈。其中精选5套不同价位的优质房源供您参考：

| 房源ID | 位置 | 户型 | 面积 | 价格 | 亮点 |
|--------|------|------|------|------|------|
| HF_331 | 国贸站 | 1室1厅 | 111.3㎡ | 1982元/月 | 近地铁（642m）、大户型、有电梯 |
| HF_18 | 望京站 | 3室1厅 | 78.2㎡ | 4794元/月 | 整租、精装、三居室 |

{"message": "为您找到以下符合条件的房源：", "houses": ["HF_331", "HF_18", "HF_53", "HF_210", "HF_254"]}
```

## 🚀 后续优化方案

### 1. Token 消耗优化（优先级：高）

#### 当前消耗分析
- 问候：0 tokens（模板回复）
- 单轮查询：约 2k-5k tokens（1-2次模型调用）
- 多轮对话：约 5k-15k tokens（每轮 1-2次调用）

#### 优化方向
1. **Prompt 精简**
   - 当前 System Prompt 约 400 tokens，可进一步压缩
   - 工具描述去掉冗余说明
   - 示例改为更简洁的格式

2. **上下文压缩**
   - 当前保留完整对话历史，可以只保留最近 3 轮
   - 提取用户需求为结构化数据（district, price, bedrooms 等）
   - 工具返回结果只保留关键字段（house_id, price, district）

3. **智能缓存**
   - 同一 session 内相同查询条件，直接返回缓存结果
   - 房源详情缓存（避免重复调用 get_house_detail）

### 2. 时间片优化（优先级：高）

#### 当前时间片消耗
- 公式：t = 1 + max(0, (n-1k)) * 0.3
- 单次调用 5k tokens ≈ 2.2 时间片
- 单轮查询（2次调用）≈ 4-5 时间片
- 多轮对话（4次调用）≈ 8-10 时间片

#### 优化方向
1. **减少模型调用次数**
   - 当前：意图识别 + 工具调用 → 结果生成（2次）
   - 优化：对于简单查询，工具调用后直接返回，不做二次总结（1次）
   - 适用场景：用户明确说"查询XX房源"，不需要额外解释

2. **批量工具调用**
   - 当前：串行调用多个工具
   - 优化：并行调用独立工具（如同时查地标 + 查房源）

3. **提前终止**
   - 如果工具调用失败（API 502），直接返回错误，不做二次模型调用

### 3. 答案命中率优化（优先级：高）

#### 当前问题
- 模型可能理解错用户意图（如"两居室"被理解为"2个卧室的合租"）
- 筛选条件不够精确（如"近地铁"没有正确设置 max_subway_dist=800）
- 房源推荐不够智能（只返回前5个，没有排序优化）

#### 优化方向
1. **意图识别增强**
   - 添加意图分类（查询、对比、租房、退租、闲聊）
   - 针对不同意图使用不同的 Prompt 模板
   - Few-shot 示例覆盖常见误解场景

2. **参数映射优化**
   - "近地铁" → max_subway_dist=800
   - "地铁可达" → max_subway_dist=1000
   - "两居室" → bedrooms=2 + rental_type=整租
   - "性价比高" → sort_by=price + sort_order=asc

3. **智能排序**
   - 综合评分：(地铁距离权重 * 归一化距离) + (价格权重 * 归一化价格) + ...
   - 用户偏好学习：记录用户点击/租赁的房源特征，调整推荐权重

4. **多平台对比**
   - 同一房源在链家/安居客/58同城的价格对比
   - 自动推荐最低价平台

### 4. 鲁棒性优化（优先级：中）

#### 错误处理
1. **API 调用失败**
   - 当前：返回 error JSON，模型生成道歉文本
   - 优化：重试机制（最多3次），降级策略（使用缓存数据）

2. **模型调用超时**
   - 当前：httpx timeout=60s
   - 优化：设置更短的超时（30s），超时后返回"系统繁忙"

3. **输入验证**
   - 当前：无输入验证
   - 优化：检查 session_id 格式、message 长度限制（< 1000字符）

#### 边界情况
1. **无房源匹配**
   - 当前：返回空列表
   - 优化：放宽筛选条件重新查询，或推荐相似房源

2. **模糊需求**
   - 当前：模型自行理解
   - 优化：主动追问关键信息（"您的预算大概是多少？"）

```

│  - 调用 POST /api/v1/chat               │

              ↓
┌─────────────────────────────────────────┐
│  Agent HTTP 服务 (FastAPI)              │
│  - 接收用户消息                          │
│  - 返回 JSON 格式响应                    │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│  Agent Core (核心逻辑)                   │
│  1. 意图识别（简单问候 → 模板回复）      │
│  2. 模型调用（Function Calling）         │
│  3. 工具执行（并行调用 API）             │
│  4. 结果生成（JSON 格式）                │
└─────────────────────────────────────────┘
       ↓                    ↓
┌──────────────┐    ┌──────────────┐
│ 模型客户端    │    │ 工具层        │
│ - Kimi       │    │ - 租房 API    │
│ - qwen3-32b  │    │ - 地标 API    │
└──────────────┘    └──────────────┘
```

### 核心流程

#### 1. 简单问候（0次模型调用）
```
用户: "你好"
  ↓
_is_simple_chat() → True
  ↓
返回模板: "您好！我是租房助手..."
```

#### 2. 单轮查询（1-2次模型调用）
```
用户: "查询海淀区两居室"
  ↓
第1次模型调用（带 tools）
  ↓
模型返回: tool_calls=[{name: "search_houses", args: {district: "海淀", bedrooms: "2"}}]
  ↓
执行工具: search_houses(district="海淀", bedrooms="2")
  ↓
第2次模型调用（带工具结果）
  ↓
模型返回: {"message": "...", "houses": ["HF_1", "HF_2"]}
  ↓
提取 JSON 并返回
```

#### 3. 多轮对话（每轮1-2次调用）
```
第1轮: "我想在朝阳区找房"
  ↓
session_manager 记录: user_message + assistant_response
  ↓
第2轮: "要精装修的"
  ↓
构建消息: system + 第1轮历史 + 第2轮 user_message
  ↓
模型理解上下文 → 调用 search_houses(district="朝阳", decoration="精装")
  ↓
返回筛选后的房源
```

### 关键技术点

#### 1. Function Calling
- **格式**：OpenAI 兼容的 tools 定义
- **流程**：模型 → tool_calls → 执行工具 → 模型 → 最终回复
- **优势**：模型自动选择工具和参数，无需硬编码规则

#### 2. 会话管理
- **存储**：内存字典 {session_id: Session}
- **内容**：messages（完整对话历史）+ requirements（提取的需求）+ candidate_houses（候选房源）
- **清理**：无自动清理机制（待优化）

#### 3. JSON 提取
- **方法1**：直接 json.loads()
- **方法2**：正则提取包含 "houses" 的 JSON 块
- **兜底**：从工具结果里提取 house_id，模型输出作为 message

#### 4. 代理问题修复
- **问题**：httpx 默认使用系统代理，localhost 请求被拦截返回 502
- **解决**：AsyncClient(trust_env=False) 禁用代理
